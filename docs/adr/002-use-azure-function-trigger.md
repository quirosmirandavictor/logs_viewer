# ADR-0002: Use Azure Functions Queue Trigger

## Status

Accepted <!-- Proposed | Accepted | Rejected | Deprecated | Superseded -->

## Date

2026-04-02

## Context

The system must process a continuous stream of log messages generated by multiple applications implemented in different technologies, including .NET (NLog) and Python using standard logging libraries.

ADR-0001 already established Azure Queue Storage as the messaging layer for the ingestion pipeline.

This ADR selects the event-driven mechanism that reacts to messages arriving in that queue and starts log processing without introducing a separate polling service.

The solution must integrate natively with Microsoft Azure services, be cost-effective, and support serverless event-driven processing.

## Decision

We will use the Azure Functions Queue Trigger to detect and process log messages as they arrive in Azure Queue Storage.

The Queue Trigger was selected because it provides:

- Serverless execution.
- Automatic scaling based on queue pressure.
- Native integration with Azure Queue Storage.
- Pay-per-execution cost model.
- Minimal infrastructure management.
- Built-in retry and poison-message handling.

Compared with the evaluated alternatives, the Queue Trigger is the best fit because it keeps the system event-driven without adding a separate polling component or always-on compute.

## Alternatives Considered

### Option A: Azure Logic Apps

**Pros**

- Low coding
- Integrated connectors
- Retry capabilities

**Cons**

- Higher latency for high-volume per-message processing
- Less suitable for fine-grained log processing logic

### Option B: Azure App Service WebJobs

**Pros**

- Simple to host alongside an App Service application
- Full control over execution flow

**Cons**

- Requires always-on compute, which weakens the serverless model
- More operational overhead than a queue-triggered function

### Option C: Custom polling worker or background service

**Pros**

- Full control over message handling
- Can be tailored for custom batching or scheduling needs

**Cons**

- Must implement polling, retries, and poison-message handling manually
- Requires additional code and operational support

### Option D: Azure Functions Queue Trigger (Selected)

**Pros**

- Serverless execution.
- Automatic scaling.
- Pay-per-execution model.
- Native integration with Azure Queue Storage.
- Built-in retry and poison-message support.

**Cons**

- Processing is at-least-once, so handlers must be idempotent.
- No strict FIFO guarantee.
- Some latency can appear due to queue polling and cold starts, depending on the hosting plan.

## Consequences

### Positive

- Keeps the pipeline decoupled while preserving the decision made in ADR-0001.
- Enables asynchronous log processing.
- Reduces infrastructure and operational costs.
- Uses Azure-native event-driven processing.
- Provides a clear operational model for retries and poison messages.

### Negative / Trade-offs

- Message handlers must be idempotent because delivery is at-least-once.
- Strict ordering is not guaranteed.
- Poison queue messages still require monitoring and manual remediation.

### Items to Monitor

- Handler idempotency and message integrity.
- Message processing latency.
- Queue length during bursts.
- Poison queue growth and repeated failures.
