# ADR-0001: Use Azure Queue Storage

## Status

Accepted <!-- Proposed | Accepted | Rejected | Deprecated | Superseded -->

## Date

2026-04-01

## Context

The system must process a continuous stream of log messages generated by multiple applications implemented in different technologies, including .NET (NLog) and Python using standard logging libraries.

The ingestion layer must be resilient to traffic spikes, allow producers and consumers to evolve independently, and support asynchronous processing without blocking the applications generating the logs.

Additionally, the solution must integrate natively with Microsoft Azure services, be cost-effective, and support serverless event-driven processing through Azure Functions.

## Decision

We will use Azure Queue Storage as the messaging service for the log ingestion pipeline.

Azure Queue Storage was selected because it provides:

- Asynchronous message processing.
- Loose coupling between producers and consumers.
- Native integration with Azure Functions.
- High availability and fault tolerance.
- High scalability.
- Low operational complexity and cost.

Compared to the alternatives evaluated, Azure Queue Storage was preferred over **Azure Service Bus** because the pipeline does not require advanced enterprise messaging features (topics/subscriptions, sessions, transactions), and the cost difference is significant at our expected volume. It was preferred over **RabbitMQ** because it is a fully managed service, avoiding the operational overhead of provisioning, patching, and maintaining a self-hosted broker, which the team is not staffed to support.

## Alternatives Considered

### Option A: Azure Service Bus

**Pros**

- Ideal for complex enterprise systems
- Support for bigger messages than Azure Queue Storage
- Unlimited time retention

**Cons**

- It is more expensive
- Max number of queues limited

### Option B: RabbitMQ

**Pros**

- Flexibility
- Excellent Performance dealing with complex messages

**Cons**

- Isn't an entirely managed service
- The team is fully responsible for maintenance

### Option C: Azure Queue Storage (Selected)

**Pros**

- Asynchronous processing.
- Loose coupling between producers and consumers.
- Native Azure Functions integration.
- High availability and fault tolerance.
- Cost-effective.
- High scalability.

**Cons**

- Limited messaging features compared to Azure Service Bus.
- Maximum message size of 64 KB.
- No built-in ordering guarantees.

## Consequences

### Positive

- Improves system decoupling.
- Enables asynchronous log processing.
- Reduces infrastructure and operational costs.
- Supports horizontal scaling of producers and consumers.

### Negative / Trade-offs

- Limited messaging capabilities compared to Azure Service Bus.
- Retry and dead-letter strategies must be implemented where necessary.

### Items to Monitor

- Queue length during peak traffic.
- Message processing latency.
- Storage transaction costs as workload grows.
